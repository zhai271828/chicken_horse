## 二、架构说明

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        Cloudflare Pages                         │
│                     (静态前端托管，免费)                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐        │
│  │ 玩家A    │  │ 玩家B    │  │ 玩家C    │  │ 玩家D    │        │
│  │ 浏览器   │  │ 浏览器   │  │ 浏览器   │  │ 浏览器   │        │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘        │
└───────┼──────────────┼──────────────┼──────────────┼──────────────┘
        │              │              │              │
        │         WebSocket 连接       │              │
        ▼              ▼              ▼              ▼
┌─────────────────────────────────────────────────────────────────┐
│              Cloudflare Workers + Durable Objects                │
│                        (游戏服务器)                              │
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                  GameRoom (Durable Object)               │   │
│  │                                                         │   │
│  │  - WebSocket 连接管理                                    │   │
│  │  - 游戏状态维护                                          │   │
│  │  - 物理模拟                                              │   │
│  │  - 碰撞检测                                              │   │
│  │  - 状态广播                                              │   │
│  └─────────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────────┘
```

### 2.2 数据流

```
玩家输入 → NetworkManager.sendInput() → 服务器接收
    ↓
服务器处理输入 → 物理模拟 → 碰撞检测
    ↓
服务器广播状态 → NetworkManager.on('GAME_STATE')
    ↓
NetworkRunState 渲染
```

### 2.3 消息流程

```
1. 创建房间
   客户端: CREATE_ROOM { playerName, character }
   服务器: ROOM_CREATED { roomId, playerId, players }

2. 加入房间
   客户端: JOIN_ROOM { roomId, playerName, character }
   服务器: ROOM_JOINED { roomId, playerId, players }
   广播: PLAYER_JOINED { playerId, playerName, character }

3. 游戏状态同步
   服务器: GAME_STATE { stage, players, obstacles, coins, scores, timeLeft }

4. 玩家输入
   客户端: PLAYER_INPUT { left, right, jump }

5. 放置陷阱
   客户端: PLACE_OBSTACLE { type, x, y, direction }
   广播: OBSTACLE_PLACED { ... }
```

## 三、部署指南

### 3.1 部署 Workers

```bash
# 1. 安装 wrangler CLI
npm install -g wrangler

# 2. 登录 Cloudflare
wrangler login

# 3. 进入 workers 目录
cd workers

# 4. 安装依赖
npm install

# 5. 部署
wrangler deploy
```

部署后会得到一个 Worker URL，类似：
`https://chicken-horse-game.your-subdomain.workers.dev`

### 3.2 配置前端

1. **创建环境变量文件** `.env`:
   ```
   VITE_WORKER_URL=wss://chicken-horse-game.your-subdomain.workers.dev
   ```

2. **代码集成已完成** ✅

   以下文件已修改完成：
   - `src/config/GameStage.js` - 添加了 LOBBY、NETWORK_RUN 阶段
   - `src/sketch.js` - 添加了 NetworkManager 和新状态注册
   - `src/ui/SplashScreen.js` - 添加了"联机"按钮
   - `src/states/MenuState.js` - 处理联机按钮点击事件

### 3.3 部署前端

```bash
# 构建
npm run build

# 部署到 Cloudflare Pages
# 方法1: 通过 GitHub 连接（推荐）
# 方法2: 直接上传 dist 文件夹
```

   - 掌握了 WebSocket 在 Workers 中的用法

2. **游戏网络同步**
   - 实现了状态同步机制
   - 学习了位置插值技术
   - 理解了客户端-服务器架构

   - 创建了独立的网络模块
   - 实现了事件驱动的消息处理
   - 提高了代码可维护性

### 8.3 经验教训

1. **服务器权威**
   - 所有游戏逻辑应该在服务器处理
   - 客户端只负责输入和渲染
   - 防止作弊，保证公平

2. **网络延迟**
   - 需要客户端预测减少延迟感
   - 位置插值让其他玩家移动平滑
   - 输入节流减少带宽占用

3. **错误处理**
   - 网络连接可能随时断开
   - 需要完善的重连机制
   - 用户友好的错误提示
