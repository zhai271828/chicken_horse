Chicken Horse 项目联机功能分析报告

目录：E:\code\chicken_horse。审查日期：2026-05-14。范围：前端状态机、网络层、Cloudflare Worker、文档和现有测试。

可编译npm run build 通过，Worker wrangler deploy --dry-run 通过。
未闭环联机流程跑酷后不能稳定进入商店、布置、下一轮。
38/38现有单元测试全过，但没有覆盖联机状态机和 Worker 协议。

一、结论

联机功能一直无法真正完成的核心原因：当前项目同时存在“Host 客户端权威”和“Worker 房间服务端管理”两种思路，但两边没有形成闭环。Host 客户端负责跑物理和结算，Worker 又负责房间、钱包、库存、商店、布置阶段；然而 Host 的结算结果没有写回 Worker，Worker 也没有在收到回合结束后推进自己的状态。

因此现在的联机只能算“大厅 + 第一轮跑酷同步雏形”。它不是一个完整的多人游戏循环。第一轮跑完后容易卡在结算；即使强行进入商店/布置，也会丢数据；布置过陷阱后再进入跑酷，Host 本地 RunState 需要真实障碍物实例，但网络布置阶段保存的是普通 JSON 对象，物理循环会崩。

三、联机流程断点

1. 大厅Worker 接收连接并发 WELCOME。客户端随后创建/加入房间。
2. 开始游戏Host 触发 START_GAME，Worker 广播 GAME_STARTED。
3. 跑酷Host 本地跑 RunState，其他客户端只接收状态。
4. 结算Host 发送 ROUND_ENDED、ROUND_RESULTS、STAGE_CHANGE。
5. 商店/布置应由 Worker 推进，但当前不会可靠发生。

代码证据：src/states/NetworkRunState.js:163-215 中 Host 自己发结算和切换；workers/src/GameRoom.js:231-237 对 ROUND_ENDED、STAGE_CHANGE、ROUND_RESULTS 只做 relay；workers/src/GameRoom.js:450-482 的 endRound() 虽然会启动商店，但没有调用入口。

四、主要 Bug 与风险

P0 Worker 收到回合结束只转发，不更新房间状态，也不调用 endRound()。 结算后不会稳定进入商店，游戏循环断掉。 workers/src/GameRoom.js:231-237、workers/src/GameRoom.js:450-482

P2 Worker 状态只存在内存字段，没有 Durable Object storage 持久化。 本地测试不明显，但线上 Durable Object 重启/迁移后房间状态可能丢失。 workers/src/GameRoom.js:38-48、workers/src/GameRoom.js:995-997

五、为什么“看起来写了很多，但还是联不了”

  1. 协议和状态没有单一真相。玩家、分数、钱包、库存、障碍物到底以 Host 本地为准，还是以 Worker 为准，目前没有统一。
  2. 阶段推进靠消息碰运气。多个状态依赖“先收到消息再注册处理器”的反向时序，导致商店物品和库存很容易丢。
  3. 本地版对象不能直接网络序列化。本地障碍物是带方法的类实例，网络发的是 JSON；从 JSON 回到可参与物理的实例缺少工厂/反序列化层。
  4. 后端没有测试兜底。现在测试全是本地系统单测，没有“两个客户端完成一整轮”的集成测试，所以流程断点不会被自动发现。
  5. 文档和代码不同步。联机实践.md 写了 VITE_WORKER_URL 配置方案，但代码没有读取；游戏设计.md 中有些已修复项和未修复项混在一起，容易误判进度。

六、还需要完成的事情

第一阶段：把联机架构收敛

  • 选定权威模型。建议短期采用 Host-authoritative + Worker relay，因为当前 RunState 物理和障碍物类都在客户端，迁移到 Worker 成本更高。
  • Worker 不再拥有独立分数和钱包计算，只维护房间、连接、阶段、已确认的结算数据、库存数据。
  • Host 在回合结束发送一个权威 ROUND_COMMIT，Worker 校验发送者是 Host 后保存结果并广播，再统一进入 NETWORK_RESULTS、NETWORK_SHOP。

第二阶段：修复阶段数据传递

  • 所有阶段启动消息都要写入 ctx.pendingNetworkShop、ctx.pendingNetworkBuild 这类缓存，目标状态 enter() 先读缓存。
  • NetworkManager 增加最近一次消息缓存或 request/response 机制，避免 WELCOME、SHOP_PHASE_START 这类一次性消息丢失。
  • 把 VITE_WORKER_URL 接到 ctx.workerUrl，生产环境自动使用 wss://。

第三阶段：网络障碍物反序列化

  • 从 BuildState._createObstacle() 抽出公共 ObstacleFactory，让本地 Build 和 Network Build 共用。
  • 网络传输只传 { id, type, gridX, gridY, direction, placedBy, state }，Host 进入跑酷前转换成真实障碍物实例。
  • 补齐特殊状态：传送门配对、炮台/风向方向、炸弹状态、Shadow 轨迹、移动平台路径等。

第四阶段：服务端校验和同步

  • Worker 校验购买价格、钱包、库存、放置阶段、放置范围、重复位置。
  • 发送者放置陷阱时不要重复添加；要么完全等 Worker 确认再落地，要么使用相同 id 合并确认。
  • 撤销、橡皮擦、地图切换都必须有协议消息，并由 Worker 广播最终结果。

第五阶段：测试

  • 新增 Worker 协议单测：创建房间、加入、开始、提交结算、进入商店、购买、进入布置、放置、回到跑酷。
  • 新增两个浏览器的 Playwright 集成测试，至少覆盖“一整轮完整闭环”。
  • 新增障碍物序列化/反序列化测试，确保每种陷阱从网络数据恢复后能被 RunState 正常 update() 和 draw()。

七、建议修复顺序

顺序 任务 完成标准
1 删除/隔离 server.js demo 路线，明确 README 只讲 Worker 联机。 开发者不会误以为 node server.js 能跑 Vite 游戏联机。
2 修复 Worker URL 配置和 WELCOME race。 本地和部署环境都能稳定创建/加入房间。
3 实现 Host 提交结算，Worker 保存并推进商店。 第一轮跑完后所有客户端稳定看到同一份结算和同一批商店物品。
4 修复 SHOP_PHASE_START、BUILD_PHASE_START 的 pending 数据缓存。 商店物品、钱包、库存不会因状态切换丢失。
5 抽出障碍物工厂，把网络障碍物转换为真实实例。 布置任意陷阱后，下一轮 Host 跑酷不报 obs.update is not a function 类错误。
6 补协议测试和双客户端集成测试。 CI 能自动验证完整联机循环。

九、最终判断

当前项目的基础质量并不差，本地游戏系统和资源量都已经不少；联机失败主要是多人架构层没有定责。最小可行路线不是继续在各个状态里补 if/else，而是先把“谁拥有权威状态”定下来，然后把回合提交、商店、布置、障碍物反序列化这四个闭环打通。

建议短期目标：先实现 2 人、固定 Host、不支持断线恢复的一整轮闭环。等这个链路稳定后，再扩展 3-4 人、重连、服务端权威物理和公网部署。
