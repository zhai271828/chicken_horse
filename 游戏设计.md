# 方案评估与优化建议

## 一、进化系统评估

### 1.1 问题分析

**滚雪球效应风险：**
- 当前设计："第一个到达终点才能进化"
- 问题：赢家越来越强，输家永远无法进化，体验极差
- 这会导致"强者恒强"，新手或落后玩家完全没有游戏体验

### 1.2 建议修改方案

**方案A：到达即进化（推荐）**
```
无论第几名，只要到达终点就能进化
- 第1名：进化 + 额外金币奖励
- 第2名：进化 + 少量金币
- 第3/4名：进化
```
优点：保证所有玩家都能体验进化，保持竞争性

**方案B：积分制进化**
```
每次到达终点获得积分，积分累积到阈值自动进化
- 第1名：3分
- 第2名：2分
- 第3/4名：1分
- 进化需要：3分/6分/10分
```
优点：更平滑的成长曲线

# 修改报告

## 修复内容总览

### 1. 开始界面方框问题 ✅ 已修复
**问题**：`>>> 按空格键开始游戏 <<<` 中间的中文显示为8个空白方框
**原因**：`homepageFont` 是 `PanasChill` 字体，不支持中文字符
**修复**：强制使用 `Noto Sans SC` 字体显示中文，改为 `[ 按空格键开始游戏 ]`
**文件**：`src/ui/SplashScreen.js`

### 2. 粘液跳跃bug ✅ 已修复
**问题**：踩了粘液后无法跳跃，复活后W键也没反应
**原因**：`applyEffect` 直接修改了 `player.jumpVel`，导致跳跃力永久降低
**修复**：
- 添加 `_baseJumpVel` 属性到 Player.js 保存原始跳跃力
- 重写 Slime.js 的 `preEffect` 方法，使用临时乘数而非直接修改
- 效果结束后自动恢复原始跳跃力
- 不可叠加（同一时间只保留一个效果）
**文件**：`src/entities/obstacles/Slime.js`, `src/entities/Player.js`

### 3. 传送蘑菇无效果 ✅ 已修复
**问题**：踩了传送蘑菇后没有实际传送
**原因**：检测空地时只检查 `MAP[ty][tx] === 0`，但实际地图使用 `TileType.SOLID` 等字符串
**修复**：
- 支持多种瓦片类型表示（0, 'empty', 'EMPTY', TileType.EMPTY）
- 地面检测也支持多种solid表示
- 添加备用方案：找不到空地时传送到地图上方随机位置
**文件**：`src/entities/obstacles/MushroomTeleporter.js`

### 4. 激光穿越障碍物 ✅ 已修复
**问题**：激光可以穿越方块，瞄准时间太短
**原因**：没有检测光束路径上的固体瓦片
**修复**：
- 瞄准时间从1秒增加到2秒
- 添加光束路径检测：逐点检查是否遇到固体瓦片
- 遇到固体瓦片时，光束在该点截止
- 导入 TileType 用于瓦片类型判断
**文件**：`src/entities/obstacles/Laser.js`, `src/config/GameConfig.js`

### 5. 火焰范围太小 ✅ 已修复
**问题**：火焰只有上面两格，杀伤力太低
**原因**：使用矩形检测范围，只向上延伸
**修复**：
- 改为圆形检测范围，以火焰为中心2格半径
- 周围一圈两格内都会被火烧
- 使用距离计算替代矩形碰撞检测
**文件**：`src/entities/obstacles/Flame.js`

### 6. 滚动刺球只滚动一次 ✅ 已修复
**问题**：滚动刺球只会滚动一次就停止
**原因**：`_onGround` 检测过于严格，只检查正下方一格
**修复**：
- 扩大地面检测范围：检查下方2格内是否有固体
- 添加 `isSolidTile` 辅助函数支持多种瓦片类型
- 确保刺球持续滚动，碰墙后掉头
**文件**：`src/entities/obstacles/SpikedBall.js`

### 7. 弓箭/炮弹力度固定 ✅ 已修复
**问题**：弓箭和炮弹速度固定，玩家容易预测轨迹
**原因**：使用固定的 `ARROW_SPEED` 和 `CANNON_PROJECTILE_SPEED`
**修复**：
- 弓箭：随机速度 = 基础速度 × (0.5~1.0)，实现3-6格随机距离
- 炮弹：随机速度 = 基础速度 × (0.8~1.2)，增加不可预测性
**文件**：`src/entities/obstacles/Arrow.js`, `src/entities/obstacles/Cannon.js`

### 8. 传送门弹飞力度低 ✅ 已修复
**问题**：传送后弹飞力度太低
**原因**：随机力度范围是3-8，向上偏移只有-3

# Cloudflare 部署联机方案

## 一、当前架构分析

### 现状
- **前端**：p5.js + Vite，纯客户端渲染
- **游戏逻辑**：全部在浏览器端运行
- **多人模式**：本地同屏，共享键盘
- **数据存储**：无服务器，无持久化

### 联机需要解决的核心问题
1. **状态同步**：两个玩家的游戏状态需要实时同步
2. **输入同步**：键盘输入需要发送到服务器
3. **权威服务器**：防止作弊，服务器决定游戏逻辑
4. **房间系统**：创建/加入房间，匹配玩家

---

## 二、技术方案选择

### 方案A：WebSocket + Cloudflare Durable Objects（推荐）
**架构**：
```
玩家A浏览器 ←→ Cloudflare Workers (WebSocket) ←→ Durable Objects ←→ Cloudflare Workers (WebSocket) ←→ 玩家B浏览器
```

**优点**：
- 延迟低（WebSocket长连接）
- Durable Objects提供有状态的房间管理
- Cloudflare全球边缘网络，延迟低
- 无需自己管理服务器

**缺点**：
- 需要重构游戏逻辑为客户端-服务器架构
- Durable Objects有学习曲线
- 免费额度有限（每天1000请求）

### 方案B：WebRTC P2P + Cloudflare Workers（信令）
**架构**：
```
玩家A浏览器 ←→ WebRTC P2P ←→ 玩家B浏览器
                ↑
        Cloudflare Workers（信令服务器）
```

**优点**：
- P2P直连，延迟最低
- 服务器成本低（只做信令）
- 适合2-4人小规模联机

**缺点**：
- NAT穿透可能失败
- 需要STUN/TURN服务器
- 实现复杂度高

### 推荐：方案A（WebSocket + Durable Objects）
